# KRAVIONA CRM — Missing Backend Endpoints

# Source: API_GUIDE.md (v2.0) cross-checked against every CRM page in this admin panel.

# Every page below has been migrated to call the live API via src/services/api.js.
# If a backend route is missing or returns 404, the page falls back to its seeded

# 1.1 Content Decay dashboard expects a richer payload than /api/posts
#      Page:      /dashboard/content-decay  →  ContentDecay.jsx
#      Currently: fakes the report from the regular /posts list
#      Needs:     GET /api/posts/decay-report
#                 ? Returns per-post: { _id, title, slug, lastReviewedAt,
#                                       views (last 30/60/90 days), decayScore,
#                                       status: "Fresh" | "Aging" | "Stale" | "Dead" }
#      Optional:  PUT /api/posts/:id/reviewed
#                 ? Currently the page calls /api/v1/blog.update(...)
#                   with { lastReviewedAt }, which is not a documented field.

# 1.2 Lead / Pipeline dashboard has rich kanban & conversion metrics
#      Page:      /dashboard/leads  →  Leads.jsx
#      Currently: maps /api/admin/messages onto the Lead view-model
#      Needs (optional but recommended):
#        GET    /api/v1/leads                       — paginated leads with stage
#        POST   /api/v1/leads                       — create
#        PUT    /api/v1/leads/:id                   — update (stage, budget, note…)
#        DELETE /api/v1/leads/:id
#        PATCH  /api/v1/leads/:id/stage             — quick stage change
#        GET    /api/v1/leads/stats                 — counts per stage + won revenue

# 1.3 Tech Stack CRUD writes to the same SiteConfig document but the spec
#      only exposes a single "techStack" array for the homepage. There is
#      no dedicated CRUD endpoint for the company-wide tools list.
#      Page:      /dashboard/tech-stack  →  TechStack.jsx
#      Currently: PUTs whole SiteConfig on every save (heavy).
#      Recommended new endpoints:
#        GET    /api/v1/tech-stack
#        POST   /api/v1/tech-stack
#        PUT    /api/v1/tech-stack/:id
#        DELETE /api/v1/tech-stack/:id
#        PUT    /api/v1/tech-stack/reorder

# 1.4 Global Settings drill-down pages each call SiteConfig on save, but
#      the schema doesn't define every sub-section used by the editor.
#      Page fields currently saved but not in the published schema:
#        /dashboard/global/contact-info   →  contactInfo { primaryPhone, whatsappLink, additionalContacts[], … }
#        /dashboard/global/address        →  address { primary{...}, additionalOffices[], showInFooter, showOnContactPage }
#        /dashboard/global/nav            →  nav { showCtaButton, links[{ label,href,isVisible,hasDropdown }] }
#        /dashboard/global/footer         →  footer { logoText, backgroundColor, textColor, showNewsletterBlock, columns[{title,links[]}] }
#        /dashboard/global/newsletter     →  newsletter { provider, mailchimpListId, doubleOptIn, welcomeEmail{...} }
#        /dashboard/global/analytics      →  analytics { gtmId, gaId, facebookPixelId, hotjarId, microsoftClarityId, customHeadScript, customBodyScript }
#        /dashboard/global/maintenance    →  maintenance { enabled, title, message, expectedReturnTime, bypassToken, allowedIPs[], redirectUrl }
#      Recommendation: extend the SiteConfig Mongoose schema to include
#      these top-level fields (see API_GUIDE.md §10.1). Without them
#      PUT /api/v1/site-config will silently drop the values.
